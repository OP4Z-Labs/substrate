- **Fixes shipped:** 1 (Cadence — global-bin symlink resolution)

### Top 3 findings

1. **Cadence (P0) — `npm link`'d binary was silently no-oping.** The
   ESM `invokedDirectly` guard compared raw paths instead of realpaths,
   so `cadence --help` produced empty output when invoked through the
   global bin symlink. Fixed in commit `3995a60`. The exact same risk
   pattern exists nowhere else (flint uses a simpler bootstrap).

2. **Cadence (P1) — `cadence knowledge refresh` only extracts the first
   docker-compose service when a block-scalar (`command: >`) appears.**
   On OP4Z's real compose file (59 services), only 1 service made it
   into KNOWLEDGE.md. Root cause is the documented limitation in
   `yaml-mini.ts` (block scalars `|` / `>` are explicitly out of scope).
   The fixture used in unit tests doesn't include `command: >`, so the
   tests still pass. NOT trivially fixable in the mini parser — needs
   either a proper YAML lib swap or hand-rolled block-scalar handling.
   Recommend v0.5: swap to `yaml` (Eemeli Aro) when the detector engine
   lands, as already flagged in v0.3 handoff open question 3.

3. **Flint (positive) — comment preservation in `wrangler.toml`
   writeback works correctly.** Added a custom marker comment, ran
   `flint add r2 ANOTHER_R2`, and the comment was still at the exact
   same position. The load-bearing v0.2 acceptance criterion holds in
   practice.

## Fixes applied

### Cadence: `fix(cli): resolve symlinks so the global bin actually runs main()`

- **File:** `src/cli.ts` (lines 230–246)
- **Commit:** `3995a60`
- **What broke:** When installed via `npm link`, the cadence binary
  is a symlink (`~/.nvm/.../bin/cadence -> ../lib/node_modules/cadence/
  dist/cli.js`). The original ESM guard `process.argv[1] ===
  fileURLToPath(import.meta.url)` compared the symlink path against
  the resolved target path — they never match, so `main()` was never
  invoked. `cadence --help` produced zero stdout / stderr with exit 0.
- **What the fix did:** Wrap both sides of the comparison in
  `fs.realpathSync`. Added a try/catch fallback so a stray broken
  symlink doesn't crash the entry detection. Includes a 6-line
  comment explaining the failure mode for future maintainers.
- **Gates after fix:**
  - `npm run build` — clean
  - `npm run lint` — clean (0 errors, 0 warnings)
  - `npx vitest run` — 84/84 tests passing
  - `cadence --version` — prints `0.3.0`
  - `cadence --help` — prints the full command list
  - `cadence audit --list` — prints the audit registry

No regressions in the test suite. No new dependencies.

## Non-trivial findings (surface, don't fix)

### P0 — none after the symlink fix

The symlink bug WAS a P0 (CLI completely no-ops on the documented
install path) but is now patched.

### P1 — Cadence YAML parser doesn't handle docker-compose's real-world shapes

- **Severity:** P1 (acceptance criterion 5 partially fails on real
  inputs)
- **Description:** `cadence knowledge refresh` against OP4Z's
  `docker-compose.yml` (59 services, multiple `command: >` block-
  scalar entries) extracts only the first service. `yaml-mini.ts`
  documents `Multi-line scalars (|, >)` as out of scope, but the
  knowledge-refresh acceptance criterion implies "works on real
  docker-compose files" — and most non-trivial composes use block
  scalars for `command:`, `entrypoint:`, multi-line env strings, etc.
- **Recommended scope:** v0.5, paired with the YAML library swap
  (already flagged in v0.3 handoff open question 3). Swap to `yaml`
  (Eemeli Aro, BSD, 100KB). Carry-cost: one new prod dep but full
  spec compliance.
- **Recommendation:** Defer to v0.5. The current behavior is
  conservative — if the parser hits something it doesn't understand,
  it silently emits a partial doc. Worse failure modes (infinite
  loop, crash) are explicitly prevented by the v0.3 fix mentioned in
  the handoff. A user running `cadence knowledge refresh` today and
  seeing only their first service will notice and file an issue —
  that's a reasonable failure mode.

### P2 — `cadence audit --list` doesn't surface the catalog

- **Severity:** P2 (UX gap, no broken contract)
- **Description:** Users have no in-CLI way to see what audits they
  CAN scaffold via `cadence add audit <name>` without reading the
  README. The 15-audit catalog is invisible to the tool's help
  surface.
- **Recommended scope:** v0.5
- **Recommendation:** Add `cadence add audit --list` (or
  `cadence audit --catalog`) that enumerates `templates/audits/`.
  Trivial implementation; meaningful UX win.

### P2 — Stub message references future version

- **Severity:** P2 (cosmetic, suggests staleness)
- **Description:** `cadence audit --type <name>` prints "...The
  detector runtime ... ships in v0.3." We're already in v0.3.
- **Recommended scope:** v0.5 (or sooner — one-line copy fix)
- **Recommendation:** Change to "...ships in v0.5." Saves the next
  smoke tester a "wait, why is it lying about its own version?"
  moment.

### P2 — Brief used `.` delimiter for `cadence add standard`

- **Severity:** P2 (papercut)
- **Description:** Brief asked for `cadence add standard
  backend.architecture` but cadence accepts only
  `backend/architecture`. The error message is good ("must be
  `<scope>/<area>` (e.g. `backend/architecture`). Got
  `backend.architecture`."), so users won't be lost — but onboarding
  docs (and the brief) should converge.
- **Recommended scope:** v0.5 or docs-only fix now
- **Recommendation:** Either accept both delimiters with normalization,
  or update README/brief to consistently use `/`. Less surface area
  is better — prefer the single-delimiter form (`/`).

### P2 — Flint `configure --dry-run` requires live Cloudflare API for token verify

- **Severity:** P2 (CI ergonomics)
- **Description:** Brief asserts dry-run "prints planned commands +
  diff without invoking anything" but in practice the token-verify
  REST call still fires. Without a valid token, dry-run aborts.
- **Recommended scope:** v0.5
- **Recommendation:** When `--dry-run` is passed, skip the verify
  call and print a banner ("Skipping CF API verify — `--dry-run`")
  so users can preview a plan offline. Acceptance criterion 5's
  intent is "no side effects" — verify is a GET, but it still
  requires creds.

### P2 — Flint v0.2 package.json still reads `version: 0.1.0`

- **Severity:** P2 (release hygiene)
- **Description:** `package.json` carries `"version": "0.1.0"`,
  so `flint --version` reports `0.1.0` even though the v0.2 handoff
  explicitly claims v0.2 shipped. Cadence's v0.3 handoff bumped to
  `"0.3.0"` (verified — `cadence --version` shows `0.3.0`).
- **Recommended scope:** Trivial — bump before any tag/publish.
- **Recommendation:** Bump `package.json` to `0.2.0` and re-link.
  Did not fix in this run because it's a release-management call
  (some teams cut versions on tag, not on completion).
