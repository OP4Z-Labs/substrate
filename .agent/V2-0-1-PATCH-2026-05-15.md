## Fix 1 — `new-service.yaml` ships with `body.md`

**Symptom:** `substrate doctor` warned "1 of N workflows lack a .body.md (new-service)" on every consumer. The reference template `templates/workflows/new-service.yaml` had no paired body file. The manifest itself was also OP4Z-specific (hard-coded FastAPI / Express stack pick), which isn't appropriate for a generic substrate reference.

**Design decisions:**

- Rewrote the manifest as a generic package-or-service scaffold: it asks for a `name` and a `stack`, points the AI at `substrate.config.json` for the repo's stack list + path conventions, runs a deterministic scaffold step (placeholder for the user's actual codegen — substrate ships the workflow shape, not the codegen), and chains into `audit-package` (not `audit-service`, since the workflow is package-or-service neutral).
- The body documents the four-block program: resolve conventions → pre-flight → deterministic scaffold (user's command) → chained audit. Calls out the three common scaffold-command shapes (repo-local script, `substrate create`, monorepo task runner) so the user knows which seam to wire.
- Joined the reference workflow set (`REFERENCE_IDS` in `tests/v2-reference-templates.test.ts`) so the discoverer + the existing body.md-presence test treat it like every other reference template. Removed the special-case test that previously excluded it from the regular reference checks.

**Files:**

- Modified: `packages/substrate/templates/workflows/new-service.yaml`
- New: `packages/substrate/templates/workflows/new-service.body.md`
- Modified: `packages/substrate/tests/v2-reference-templates.test.ts`

**Tests added:** the new-service files are now exercised by all 5 existing REFERENCE_IDS tests (ships, validates, discoverer registers, body loads, run dispatches) plus the dedicated v2-schema validation test (now also asserts the body.md exists).

## Fix 2 — Memory loader ignores `MEMORY.md` / `README.md` / `INDEX.md`

**Symptom:** the loader walked `~/.claude/projects/<encoded>/memory/*.md` and treated every file as a memory needing recommended frontmatter (`metadata.type` / `scope` / `tags`). Claude Code itself writes a `MEMORY.md` index at the root of every project's memory dir, so `substrate doctor` produced a spurious "lacks recommended substrate fields" warning on every consumer.

**Design decisions:**

- Added a `DEFAULT_MEMORY_IGNORE` constant exporting `["MEMORY.md", "README.md", "INDEX.md"]` — the three filenames Claude Code conventionally uses for navigation files in the memory dir.
- Made the ignore list configurable per repo via `substrate.config.json` `memory.ignore`. Additive on top of the defaults — repos that maintain `PROJECT-INDEX.md` etc. add their entry there without losing the defaults.
- Added an in-process `MemoryQueryOptions.ignore` for callers (workflows, doctor checks) to extend the list dynamically.
- `resolveMemoryIgnoreList` is exported so tests + future doctor checks can inspect the effective list.
- Match is on the basename only, case-insensitive (`MEMORY.md` ignores `memory.md`, `Memory.md`, etc.).
- Typed `memory: { path?, ignore? }` on `SubstrateConfig` so the union is explicit.

**Files:**

- Modified: `packages/substrate/src/v2/memory.ts` — `DEFAULT_MEMORY_IGNORE`, `resolveMemoryIgnoreList`, `readMemoryIgnoreFromConfig`, ignore filter in `readMemoriesFromDir`.
- Modified: `packages/substrate/src/util/types.ts` — added `memory` block to `SubstrateConfig`.
- Modified: `packages/substrate/tests/v2-memory.test.ts` — added a new `describe` block for the ignore list.

**Tests added (6):**

1. Skips `MEMORY.md` by default; real memories still surface.
2. Skips `README.md` and `INDEX.md` by default.
3. Matches names case-insensitively (`memory.md`, `Memory.md` both ignored).
4. Honors `memory.ignore` from `substrate.config.json` (additive on top of defaults).
5. Honors the in-process `ignore` option (additive on top of defaults + config).
6. Regression: `MEMORY.md` alongside real memories never triggers a frontmatter warning (the exact OP-1374 #2 scenario).

## Fix 3 — `auto/.substrate-manifest.json` populates entries on init

**Symptom:** `substrate init` was scaffolding ~50 files but writing the manifest at `entries: []`. This left `substrate uninstall`'s precise-removal path useless (had to fall back to the known-location pattern) and broke the v0.5 upgrade flow's three-way merge (no `contentHash` baseline to compare against).

**Design decisions:**

- Collect a `ManifestEntry` for every file `runInit` writes:
  - `copyTemplate`'s `created` list for the auto/ tree.
  - Bridge files — hashed from in-memory content (we already had the content rendered for `applyBridgeReplacements`).
  - `substrate.config.json` when `configCreated`.
  - Every file `scaffoldV2Layout` reports as created (the substrate/ tree).
- Write the final manifest atomically (`atomicWriteJsonSync`) at the end of init so a crash mid-init leaves either the empty stub OR the populated final — never a partial.
- Manifest entry shape: `{ path, templateVersion, contentHash: 'sha256:<hex>', ejected: false }`. Repo-relative paths so the manifest is portable across machines.
- Idempotency contract preserved: a pre-existing manifest is left untouched (it belongs to the user once it exists). Same shadcn rule as for scaffolded files.
- The manifest file itself is NOT tracked (would be recursive and break on every write).

**Files:**

- Modified: `packages/substrate/src/commands/init.ts` — manifest stub + collection + final write; `buildManifestEntry` + `buildManifestEntryFromContent` helpers.
- Modified: `packages/substrate/tests/init.test.ts` — replaced the "empty manifest stub" test with three regression tests for the new behavior.

**Tests added (3):**

1. Populated manifest tracking every scaffolded file — at minimum >5 entries, every entry has the v0.5 shape (repo-relative path, `sha256:` hash, templateVersion, `ejected: false`); spot-check that `substrate.config.json`, `substrate/workflows/*`, and `auto/instructions/main/audit-*` are tracked; the manifest itself is NOT tracked.
2. Hashes match on-disk file contents (the upgrade-flow load-bearing contract — verified by re-hashing `substrate.config.json` and comparing).
3. Idempotency: re-running init preserves the existing manifest byte-for-byte.

## Fix 4 — `BE-APIV-001` detector recognises FastAPI router-include prefix

**Symptom:** the v2.0.0 detector regex matched route decorators (`@app.get("/api/v1/users")`) and flagged any whose path didn't start with `/api/vN`. FastAPI's far more common pattern applies the prefix at include time:

```python
# users.py
router = APIRouter()
@router.get("/users")
def list_users(): ...

# api.py
app.include_router(users.router, prefix="/api/v1")
```

OP4Z uses this pattern exclusively, producing 535 false positives on the M8.9 smoke.

**Design decisions:**

- Two-pass walk:
  - **Pass 1** scans every Python file for `.include_router(<router>, prefix="...")` and records `<router-variable> → prefix` in a project-wide map. Keyed both by the full expression (`users.router`) and the last dotted component (`router`) so lookup is forgiving across import styles. Includes without a `prefix=` kwarg are skipped (no recorded prefix → routes use bare decorator path).
  - **Pass 2** walks route decorators, captures `<varName>` and `<path>` via named groups, resolves the effective path by joining the recorded prefix with the decorator path, and flags only when the effective path doesn't start with `/api/vN`.
- `@app.get(...)` decorators on the top-level FastAPI app variable are handled inline — `app` can't be included into another app, so no include-prefix lookup.
- Fallback: when an exact `varName` match isn't found, try `router` (the conventional default for routes in a sub-module). Keeps the false-negative window narrow without invoking heuristics that would re-introduce false positives.
- The `joinPath` helper normalises the `prefix/path` boundary so neither leading nor trailing slashes double up.
- Documented the two-file split + the project-wide map in the detector header so future maintainers don't think the per-file behavior is a bug.

**Files:**

- Modified: `packages/substrate/templates/standards/cross-cutting/detectors/be-apiv-001.mjs` — full rewrite as two-pass walk; preserves the same `ctx.finding()` shape so the audit reporter doesn't need changes.
- Modified: `packages/substrate/tests/v2-ripgrep-lookaround.test.ts` — added four regression tests after the existing BE-APIV-001 test.

**Tests added (4):**

1. Router-include with `prefix="/api/v1"` produces zero findings (the OP-1374 #4 case — was 535 false positives).
2. No prefix anywhere (include without `prefix=` kwarg) still flags routes that lack a versioned decorator path.
3. Non-versioned include prefix (`prefix="/legacy"`) flags the route AND the message reports the combined effective path (`/legacy/users`).
4. Decorator-only routes with `/api/v1` at the decorator (top-level `@app.get("/api/v1/health")`) still pass.
