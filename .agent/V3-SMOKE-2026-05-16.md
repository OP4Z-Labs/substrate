## Bugs found (logged, NOT fixed in this pass)

### Bug-1 â€” `substrate query *` commands NOT extends-aware in v3.0.0-alpha.1

**Surface:** `substrate query rules`, `substrate query standards`,
`substrate query doc-checks` all read only `<cwd>/substrate/...`,
ignoring the extends chain.

**Evidence:** scenario 8 confirmed `query rules --json` returned the
repo-local single rule (post-scenario-4 override) and did NOT include
the 9 other org rules from the file: source.

**Cross-check:** Agent E's V3-NE11 HANDOFF item 1 frames this as
"existing v2 single-root discoverers stay untouched; v3 wrappers opt
in." The CLI side of the opt-in (rewiring `runQueryRules` etc. to
call `discoverRulesAcrossExtends` when `extends` is present) was
not done in alpha.1.

**Severity:** medium â€” `substrate extends list` shows the right
content; daily users querying through `query`/`hooks list` see only
repo-local. Surface gap, not a correctness bug in the resolver.

**Suggested fix:** in `src/v2/deterministic/{query-rules,query-standards,query-doc-checks}.ts`
+ `src/v2/deterministic/hooks-command.ts`, branch on
`config.extends?.length` and call the merged discoverer. ~50 LOC
per command + a test per command. v3.0.0-beta.1 target.

### Bug-2 â€” `substrate hooks list` NOT extends-aware

**Surface:** Same root cause as Bug-1. The hooks list command shows
"No hooks discovered" against a setup whose extends source has 3 hooks.

**Severity:** medium. Same fix shape.

### Bug-3 â€” `substrate run <workflow>` NOT extends-aware

**Surface:** `substrate run org-git-review-pre` returns
`Workflow "org-git-review-pre" not found in substrate/workflows/.
Discovered: (none)` even though the extends chain provides it.

**Workaround in scenario 9:** copy the workflow file from the org
source into the consumer's repo-local `substrate/workflows/` before
running.

**Severity:** high â€” this is the headline daily-driver command for
adopters. Without it, the v3 value prop ("just declare extends and
run") is half-empty.

**Suggested fix:** in `src/v2/workflow/run.ts` (or wherever
`runV2Workflow` resolves the workflow id), use
`discoverWorkflowsAcrossExtends` instead of the single-root
discoverer. Returns the workflow with provenance attached so the
runner can log "running org-shared:tackle-task" etc.

### Bug-4 â€” `substrate audit` NOT extends-aware

**Surface:** `substrate audit` reads RULES.yaml from
`<cwd>/substrate/RULES.yaml` or `<cwd>/auto/RULES.yaml` only. Not
verified in this smoke (no audit run in scenario 8), but inferable
from the same code path that drove Bug-1.

**Severity:** high â€” `substrate audit` is the primary adoption
selling point: "org rules enforce on every PR." Until this is
extends-aware, adopters must copy the org's RULES.yaml into
repo-local (the substrate-shared pattern Beau's pitching).

**Severity adjustment:** could be left in alpha.1 if Bug-1's fix
also addresses the audit's rules loader (likely shares
`loadRules`).

### Bug-5 â€” Tarball does NOT include CHANGELOG.md

**Surface:** `tar -tzf op4z-substrate-3.0.0-alpha.1.tgz | grep -i changelog`
returns nothing.

**Root cause:** `packages/substrate/package.json.files` whitelist
covers `dist`, `templates`, `templates-history`, `schemas`,
`README.md`, `LICENSE` â€” no `CHANGELOG.md`. The CHANGELOG lives at
the workspace root (`/CHANGELOG.md`), not the package root
(`packages/substrate/CHANGELOG.md`).

**Severity:** low â€” CHANGELOG is published to GitHub releases /
npm registry pages separately. Consumers who `npm install
@op4z/substrate` won't get a CHANGELOG.md in node_modules; they
have to visit GitHub. Documented in scenario 10c.

**Suggested fix:** either (a) copy the workspace-root CHANGELOG
into `packages/substrate/` at `prepublishOnly` time and add
`CHANGELOG.md` to the `files` whitelist, or (b) maintain a
separate `packages/substrate/CHANGELOG.md` with package-only
entries. Beau's call.

**Adjusted scope:** scenario 10c was modified to verify the
**root** CHANGELOG entry rather than the tarball entry.

### Bug-6 (minor) â€” `extends clear-cache` prints to stdout in JSON-mode-style invocation

**Surface:** `substrate extends clear-cache --json` doesn't accept a
`--json` flag â€” it just prints `No extends cache at /tmp/...; nothing
to remove.` to stdout. That output would corrupt JSON parsing if the
caller assumed `--json` worked.

**Severity:** low. The smoke handles it by not parsing the output.

**Suggested fix:** add `--json` support to `clear-cache` (or
document the absence). 5-line patch in `extends-command.ts`.

## Test infrastructure conventions (locked)

- Fixtures at
  `packages/substrate/tests/smoke/fixtures/substrate-shared-fixture/`
  are the **canonical reference**. The smoke script copies them
  to `/tmp/substrate-smoke-XXXXXX/` working copies; fixtures are
  never modified by tests.
- All `/tmp/substrate-smoke-*` directories live outside the repo
  and are wiped by the script's exit trap.
- The script runs standalone (`bash tests/smoke/enterprise-smoke.sh`)
  AND via npm script (`npm run smoke:enterprise`).
- Each scenario prints `[OK] Scenario N: <name>` on pass or
  `[FAIL] Scenario N: <reason>` on fail (exit nonzero unless
  `SMOKE_FAIL_FAST=0`).
- The CI workflow re-packs the tarball each run, so PRs that touch
  the package source are validated against the freshly built
  artifact.
