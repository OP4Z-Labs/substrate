### Design decisions taken

These are the places where the plan was sketched and I had to pick.
Recording with rationale so v0.3 can override or confirm.

1. **TS 5.5 + Vitest 2.x instead of the brief's "TS 6.0 + Vitest 4.1"**
   Those versions don't exist on npm — TS latest is 5.6, Vitest latest is 2.x. I assumed the brief's version numbers were aspirational placeholders for "the latest at v0.1 ship time" and pinned to the current stable. Easy to bump in `package.json`.

2. **Placeholder substitution = plain `String.split().join()`, both content and path**
   Per the brief: no Handlebars. The init function extends this to path substitution (`templates/package-python/{{NAME_SNAKE}}/__init__.py`) which the brief didn't specify, but it's the obvious move because Python packages need a snake-cased source directory matching `pyproject.toml`'s `packages = [...]` field. The alternative was post-copy `fs.rename`, which is messier.

3. **Static Claude bridge, not yaml-driven generation**
   Per the brief's recommended approach. `cadence init --with-claude` copies `templates/bridges/claude/cadence.md` verbatim with `{{PROJECT_NAME}}`/`{{SHORT_CODE}}` substitution. The yaml-driven generation from `auto/config/commands.yaml` is a v0.3 concern. The bridge file documents this so users aren't surprised when the format changes.

4. **CLI is a thin dispatcher; no detector engine**
   Per the brief. `src/commands/*.ts` are pure functions; `src/cli.ts` is < 150 lines of commander wiring. No `lib/runtime/`, `lib/detectors/`, `lib/parsers/` — those would be premature when v0.1's `audit --type` is explicitly allowed to print "would run audit X".

5. **Idempotency via skip-if-exists, not overwrite-with-backup**
   `init` and `create` both refuse to clobber existing files; they log "skipped" instead. This matches shadcn's add-only ergonomic and is the v0.5 upgrade flow's natural precondition (user edits are sacred — only `cadence upgrade` ever modifies them, and only after diff-review). Tested in `init.test.ts` "idempotency" spec — sentinel content survives a re-run.

6. **Manifest stub is empty in v0.1, not pre-populated**
   The brief says "stub for the future upgrade flow — empty entries for v0.1". I followed this literally: the manifest has `{ schemaVersion: 1, cadenceVersion: "0.1.0", entries: [] }`. The alternative (pre-populating with hashes for every scaffolded file) was tempting but would force me to define the v0.5 upgrade semantics, which is out of scope.

7. **`cadence audit --type` prints a stub, but with structured data**
   v0.1 reads the instruction file, parses its front matter, and emits a `AuditRunReport { type, status: "stub", instructionPath, description, findings: 0 }`. `--json` makes it pipeable. This is more than the brief's minimum of "would run audit X" but still less than executing detectors — humans/agents can verify their scaffold loads correctly before the v0.3 runtime ships.

8. **`audit-instructions/` directory in `templates/` is unused**
   I created it during scaffold but ended up putting the three audit templates under `templates/init/instructions/main/` (so `copyTemplate(init, autoDir)` lands them in the right user-facing path). Left the empty dir cleaned out so it doesn't confuse v0.3.

9. **Deferred-command soft warning, not a hard error**
   Calling `cadence add foo` exits with code 2 (consumer error, not crash) and prints a one-liner pointing at the roadmap. The brief said deferred commands are out of scope; my read is "still discoverable in `--help` so users learn they exist". Easy to remove if the user prefers a stricter posture.

10. **`shortCode` auto-derived from project name when not provided**
    Defaults to first 1-3 uppercase letters. The brief mentions the `[OP-NNN]` convention but doesn't specify the inference rule. This is a low-stakes ergonomic choice — users typically pass `--short-code` explicitly anyway.

## Open questions for the user

Each has a recommended answer + one-line "why".

1. **TypeScript / Vitest version pinning.** The brief specified "TS 6.0,
   Vitest 4.1" — those don't exist on npm. I pinned TS 5.5 / Vitest 2.x.
   *Recommend:* keep as-is. *Why:* matches current stable; can bump
   when newer majors release without breaking semver.

2. **`templates/audit-instructions/` directory.** The brief listed this
   under templates but I ended up nesting the audit templates under
   `templates/init/instructions/main/` (so `cadence init` copies them
   to the right user-facing path). *Recommend:* leave the
   internal structure as-is; the public contract is "audit-*.md files
   live under `auto/instructions/main/`". *Why:* the user-facing
   contract is what AI tooling consumes; the internal template path is
   a v0.1 implementation detail.

3. **`telemetry.enabled: false` default + no first-run prompt.** Plan
   says opt-in only with a first-run prompt. v0.1 ships the field but
   no prompt. *Recommend:* defer the prompt to v0.8 per the roadmap.
   *Why:* prompts add UX surface that needs careful design (where to
   store the answer, how to revoke); not worth landing in v0.1.

4. **Path substitution beyond `{{NAME_SNAKE}}`.** I added path
   substitution to `copyTemplate` for the Python package directory.
   *Recommend:* document this as part of the template authoring guide
   when v0.3's `cadence add` ships. *Why:* the substitution rule is
   simple (plain string replace, no regex), but template authors need
   to know it exists.

5. **Deferred command UX — soft warn vs hide vs hard error.** Today
   they show in `--help` and exit 2. *Recommend:* keep them visible
   so users learn the surface. *Why:* discoverability beats minimalism
   when the gap is "feature lands in 6 weeks", not "feature never
   ships".

6. **High-stakes design fork I did NOT return on (but flagging here):**
   the runtime architecture for v0.3's detector engine. The brief said
   "you may legitimately hit forks worth returning on" but I didn't
   need a decision at v0.1 — the stub satisfies the acceptance
   criteria. *Recommend:* before v0.3 starts, design-doc the detector
   engine (probably `src/detectors/`: ripgrep wrapper, shell-command
   adapter, registry from `RULES.yaml`).
