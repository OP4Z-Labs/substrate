### Design decisions taken

1. **YAML lib pin: `yaml@2.9.0`.** Verified via `npm view yaml version`
   before installing. Recorded in package.json. No dep-graph fanout.

2. **Diff helper hand-rolled.** Rejected `diff` / `jsdiff` (60kB+ for
   one screen of UX). Pure LCS + unified-diff formatter in 170 lines
   of `src/util/diff.ts`. Output mirrors `git diff --unified=3` for
   human readability. Algorithm complexity is O(n*m) memory which is
   fine for cadence's expected inputs (200-line markdown files).

3. **`cadence upgrade` is a "degenerate three-way."** v0.5 ships the
   `current vs new-template` diff because the *original* template (at
   the recorded `templateVersion`) requires a `templates-history/`
   shipping path not yet built. The merge UX is honest about this in
   the merge-file comments. The manifest schema is already keyed on
   `templateVersion`, so v0.8 can layer the third anchor in without a
   migration. **Locked in code** at `src/commands/upgrade.ts` head
   comment.

4. **Upgrade choice set: keep / take-new / merge / eject.** Not
   accept-edit / abort / theirs / ours. The wording mirrors what users
   want to *do*, not what git calls them. `merge` writes a sidecar
   `.cadence-merge` file (not stomping the user's copy) so resolution
   is an editor-driven step rather than a CLI step.

5. **Plugin contract shape: tiny verb set, structural typing.**
   TaskAdapter has 5 methods; VcsAdapter has 5. No callbacks, no
   batching, no streaming. The interface is checked at load time by
   `isTaskAdapter()` / `isVcsAdapter()` guards — structural typing
   over `instanceof` so an adapter author isn't forced to import a
   cadence base class. **Locked in `src/extensions/*-adapter.ts` head
   comments.**

6. **`extensions.taskAdapter: string | null`, lazy `import()`.**
   Dynamic import keeps adapter SDK deps out of cadence's runtime
   install footprint. Null means "no adapter" for taskAdapter (and
   `cadence task` exits with an install hint). Null means "use built-
   in git" for vcsAdapter (cadence ships a default for the SCM that
   100% of users have, but stays neutral on trackers).

7. **Stub adapter at `packages/adapter-stub/` with its own
   tsconfig.** Separate build because the adapter is a peer of
   cadence, not a downstream consumer. The stub duplicates the
   interface inline (no `import` from cadence) — structural typing
   enforces compatibility at runtime. Real adapters (`@cadence/adapter-
   linear` etc.) follow this exact pattern.

8. **Workflow YAML schema locked for v0.5.** Three step types
   (command / audit / prompt). Three step strings (command / audit /
   prompt). One `var` field for prompt-answer storage. One `condition`
   field. Anything more is a v0.8 additive change. **Locked in
   `src/commands/workflow.ts` head comment.**

9. **Variable substitution: two modes.** In command/prompt strings,
   unknown vars are left as-is (typos surface loudly in the spawned
   shell). In conditions, unknown vars resolve to empty (so
   `condition: "${FLAG}"` is truthy iff FLAG was actually provided).
   This split was discovered when the first condition test failed —
   without it conditions are always-truthy. Recorded in
   `substituteVars()` head comment.

10. **Command-step trust model.** `spawnSync(cmd, { shell: true })`.
    The schema's `command` field is workflow-author content, not user
    input — same trust model as Make targets or npm scripts. Made
    explicit in the runCommandStep comment.

11. **Cursor bridge directory assumption: `.cursor/commands/cadence.md`.**
    Cursor's slash-command spec isn't fully nailed down in public
    docs. Default mirrors the Claude convention (and the broader
    VSCode-extension pattern). Header comment in
    `templates/bridges/cursor/cadence.md` documents the assumption so
    a future regeneration corrects it without breaking the dispatch
    contract.

12. **Multi-bridge precedence: `--bridge` wins over `--with-claude`.**
    `resolveBridges()` is the single source of truth. Legacy alias
    preserved for back-compat with the v0.1+ CLI signature. Unknown
    bridge names throw with the available list — silent no-op was
    rejected as too confusing.

13. **`runAuditType` now accepts `quiet` option.** Workflow audit
    steps would otherwise drown the workflow output in audit stub
    chatter. The flag is additive and defaults false so existing
    callers are unchanged.

14. **Required-imports lint rule: no `require()`.** Two early commits
    used `require('node:fs')` inside test bodies for one-off
    `unlinkSync` calls. The lint config flagged them; converted to
    top-of-file `import` statements. Pattern established for future
    test code.

## Open questions for the user

1. **Cursor bridge directory location.** Defaulted to
   `.cursor/commands/cadence.md` based on the Claude parallel +
   VSCode-extension community convention. If Cursor's docs land on
   a different path (e.g. `.cursor/prompts/`, `.cursor/rules/`), the
   correction is a one-line edit in `bridgeTargetDir()` plus a
   regen of `templates/bridges/cursor/cadence.md`'s header comment.
   *Recommend:* ship as-is for v0.5; flip when Cursor's spec
   stabilizes. *Why:* the dispatch contract (what the bridge tells
   the agent to run) is bridge-location-agnostic.

2. **Upgrade "three-way" → real three-way.** v0.5 ships the
   `current vs new` diff with a "v0.5 limitation" note. Adding the
   third anchor requires shipping `templates-history/<version>/...`
   in the cadence package. *Recommend:* defer to v0.8 since it's a
   ship-size concern (every published version inflates the package
   bundle by the templates delta). *Why:* the v0.5 UX is genuinely
   useful for the common case — most users haven't reached the
   point where they care which-version-of-the-original they're
   diffing against.

3. **Stub adapter publication.** `@cadence/adapter-stub` is named
   like an npm package but ships in this monorepo. *Recommend:*
   defer npm publication to v0.8 alongside the cadence package
   itself; the brief says cadence stays local at v0.5. *Why:* once
   we publish, version-coupling concerns (does adapter@0.5 work
   with cadence@0.5?) need a story; better to settle that during the
   pre-1.0 hardening pass.

4. **Workflow step extensions in v0.8.** The locked v0.5 schema
   covers 80% of needs (shell command, audit, ask user). Likely v0.8
   additions: `parallel` (run multiple sub-steps concurrently),
   `file` (write a file from a template), `task` (call the loaded
   TaskAdapter). *Recommend:* track in roadmap; let real users drive
   which one ships first. *Why:* speculative type proliferation is
   the failure mode of workflow runtimes.

5. **`cadence task` defaults.** OP4Z's `./exc api` enforces required
   classification (type, priority, category, complexity) on every
   create call. The v0.5 surface accepts them as flags but doesn't
   *require* them — different adapters care about different
   classifications. *Recommend:* keep loose at the cadence layer;
   let adapters reject incomplete inputs at their layer. *Why:* a
   public CLI should be the lowest-common-denominator; specific
   trackers can enforce more.

6. **Manifest schema bump cadence.** Current `schemaVersion: 1`.
   v0.5 didn't need a bump (all changes additive in the entries' shape
   were already supported). A bump becomes necessary when we want to
   add fields like "first-seen-at" or "original-content-blob". *Recommend:*
   wait until there's an actual need; never bump speculatively. *Why:*
   schema bumps need migrations; they're real work.

### Adapter contract patterns established
- **Plugin interface lives in `src/extensions/<name>-adapter.ts`.**
  Each ships an interface, an `is<Name>Adapter()` guard, and is
  imported by the loader. Adding a third plugin axis (e.g.
  `notificationAdapter`) follows this pattern verbatim.
- **Built-in default adapters in `src/adapters/<name>.ts`.** Today
  only `git.ts`. A hypothetical `mercurial.ts` would be drop-in.
- **Loader pattern:** read config → resolve null/string → dynamic
  `import(packageSpec)` → guard with `is<Name>Adapter()` → throw
  `AdapterLoadError` with actionable hint on failure. Reusable for
  any future plugin axis.

### Workflow runtime patterns established
- **YAML schema additive-only post-v0.5.** Adding new step types is
  additive (new `type` value). Adding new step fields is additive.
  Removing or renaming fields is a major.
- **Test seam:** `WorkflowStartOptions.resolvePrompt` swaps inquirer
  for a deterministic test function. Mirrors the
  `UpgradeOptions.resolveChoice` pattern. Use the same shape for any
  future interactive command (`resolve<Verb>` returns the answer
  promise).
- **Variable substitution split:** literal-preserving for
  command/prompt strings (typos surface in shell errors);
  empty-on-unknown for conditions (so falsy means "var wasn't set").
  Don't merge these — they answer different questions.

### Upgrade flow patterns established
- **State enum:** unmodified / modified / missing / ejected /
  template-gone. The classifier in `classifyEntry()` is exhaustive
  and pure (no writes). Tests pin every state.
- **Three-way is degenerate in v0.5.** When v0.8 ships
  `templates-history/<version>/`, the classifier grows a third anchor
  parameter (the original-at-templateVersion content). The
  classification states stay the same; the diff renderer grows a
  three-pane view.

### Multi-bridge precedence
- `resolveBridges()` is the single source of truth. `--bridge` wins
  over `--with-claude`. The next bridge target (MCP) drops in by:
  (a) adding to the `BridgeName` union, (b) adding a template at
  `templates/bridges/mcp/`, (c) adding a `bridgeTargetDir()` case,
  (d) extending `checkBridge()` in doctor. Tests follow
  `tests/integration/bridges.test.ts` shape.

### Patterns to NOT use
- **`require()` in test files.** ESLint rejects them. Top-of-file
  `import` only. If you need a `node:fs` function, add it to the
  existing destructure at the top of the file.
- **`process.exec` / `execSync` for git operations.** Security hook
  flags these; use `spawnSync('git', [...args], { cwd })` instead.
  Centralized in `src/adapters/git.ts` `runGit()`.

### Knowledge test for OP4Z compose
- The new test in `tests/integration/knowledge.test.ts` gates on
  `existsSync('~/dev/TheNexusProject/docker-compose.yml')`.
  In CI without the OP4Z clone, the test self-skips via
  `it.skipIf(!OP4Z_COMPOSE_AVAILABLE)`. If you move the cadence
  repo, set `OP4Z_COMPOSE_PATH` env var or the test will be silent.

### Stub adapter build
- The stub at `packages/adapter-stub/` has its OWN tsconfig and
  build. The integration `global-setup.ts` builds it before tests
  run. If you delete `packages/adapter-stub/dist/`, expect
  `tests/adapters.test.ts` to fail with a missing-module error
  pointing at `STUB_ADAPTER_PATH`.
