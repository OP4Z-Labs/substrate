### Design decisions taken

1. **`templates/audits/` parallel to `templates/init/instructions/main/`.**
   The v0.1 trio of audits lives in `templates/init/instructions/main/`
   (consumed by `cadence init`). v0.3 needs a flat catalog for
   `cadence add audit <name>` to pick from. I copied the three v0.1
   audits into `templates/audits/` so the catalog is complete. Net
   cost: 3 duplicated files. Alternative: have `cadence add audit`
   reach into the init template tree. I picked duplication because
   the templates *are* the same content right now but will likely
   diverge in v0.5 when init becomes opinion-lite and add becomes
   the primary surface. Easy to dedupe later.

2. **Standards docs ship as TODO-stub bodies, not blank.** The brief
   said "the SHAPE ships; the depth comes per-team". I interpreted
   "shape" as section headers + a few worked examples + the
   forbidden-pattern list (because the forbidden list is what catches
   real teams in real reviews). Each doc is 40-110 lines ŌĆö enough to
   be useful as a starting point, not so much that it papers over
   per-team decisions.

3. **`templates/audits/` paths are flat (`audit-<name>.md`), but
   `templates/standards/` is nested (`<scope>/<area>.md`).** Matches
   how they're consumed: audits get scaffolded into a single
   `auto/instructions/main/` directory, standards into
   `auto/standards/<scope>/<area>`.

4. **Manifest entries are append-on-write, dedup on path.** Every
   `add` call regenerates the entry for that path (in case the
   template version changed) but never deletes other entries. The
   `ejected` flag is preserved across re-records so v0.5's upgrade
   flow can still honor user opt-outs.

5. **No real YAML library dependency.** I wrote a 230-line parser
   that handles the docker-compose / config-file subset. Brief
   recommended this for v0.3; v0.5's RULES.yaml execution may need
   the full surface and can swap in `yaml`. The parser passes its
   unit tests and survived dogfooding on a multi-service compose.

6. **YAML parser bug fix: strings with colons in list items stay
   scalars.** First version of `parseSequence` treated `- ./app:/app`
   as a mapping, looped forever, OOM'd. Fix: `looksLikeMapKey` uses
   the same `findUnquotedColon` rule as `parseMapping` (colon must
   be followed by space or EOL). Test pins it.

7. **`doctor` reports use severity + fix-string rather than just
   pass/fail.** Each failed check carries a one-line fix suggestion
   ("Run `cadence init`", "Edit cadence.config.json stacks"). Makes
   the report directly actionable.

8. **Knowledge redaction is key-substring matching, not value-based.**
   `KEY` in env var key ŌåÆ redacted. This catches `SECRET_KEY`,
   `API_TOKEN`, `DB_PASSWORD` without trying to ML-detect secrets in
   values. Configurable via `cadence.config.json` ŌåÆ `knowledge.redactPatterns`.

9. **Detector severity model lives in RULES.yaml frontmatter.** Each
   rule entry in `templates/standards/cross-cutting/RULES.yaml`
   declares its severity (critical/high/medium/low) and detector
   type. v0.3 ships the schema; v0.5's runtime executes against it.

10. **`cadence add command` writes a stub, not a copy.** Unlike audit
    and standard, there's no library of pre-baked command docs that
    apply across projects ŌĆö every command surface is project-specific.
    The stub gives users the frontmatter shape and the section
    headers; the body is theirs.

11. **`cadence add scaffold/workflow` registers via YAML, not by
    copying.** The scaffold *template* lives in node_modules and is
    consumed by `cadence create`. What gets recorded in the user repo
    is the *registration* (so `cadence doctor` and v0.5's `cadence
    config --list` can introspect what's enabled). Workflows ship
    a registry entry plus a doc stub because the doc is where the
    workflow steps get authored.

12. **Stack detection falls back to python+typescript when no markers
    found.** Required to preserve v0.1 test behavior (some smoke
    flows run in empty tmp dirs). The fallback shows `(fallback
    default)` in the init log so users know to investigate.

## Open questions for the user

1. **Audit catalog dedup.** The v0.1 trio (`pre-merge`, `dependencies`,
   `dead-code`) exists in both `templates/init/instructions/main/` and
   `templates/audits/`. *Recommend:* leave duplicated for v0.3; in v0.5
   collapse to one location (`templates/audits/`) and have `init` copy
   from there. *Why:* simplifies the catalog model.

2. **Standards-doc body depth.** v0.3 ships TODO-stubs (~50-100 lines
   each). A future tooling pass could let users `cadence add standard
   --depth full` to get a more opinionated body when they want it.
   *Recommend:* defer to v0.5 when there's a real upgrade flow that
   can refresh content. *Why:* avoids the maintenance burden of
   shipping a "rich" body that's still aspirational.

3. **YAML parser swap.** v0.3's mini parser handles docker-compose
   and config files. v0.5's RULES.yaml execution may need a richer
   parser. *Recommend:* swap to `yaml` (the Eemeli Aro package) when
   the detector engine lands. *Why:* one place to make the change;
   the mini parser keeps the v0.3 surface dep-light.

4. **`add scaffold` registry format.** I picked a minimal YAML schema
   (`scaffolds: [name1, name2]`). If `cadence create` needs to read
   per-scaffold metadata (template version, custom args), the
   schema will grow. *Recommend:* extend to keyed maps when needed.
   *Why:* the v0.3 shape is the minimum for `cadence doctor` to
   verify the registration is consistent.

5. **Doctor exit codes.** Today: ok ŌåÆ 0, warn ŌåÆ 0, error ŌåÆ 1.
   Some teams want warnings to gate too. *Recommend:* add a
   `--strict` flag in v0.5 that promotes warnings to errors.
   *Why:* default is forgiving; teams that need stricter behavior
   can opt in.

6. **`cadence doctor` tooling checks.** v0.3 only checks node version
   (which is trivial ŌĆö we're already running on node). A real check
   would look for required tools per stack (`ripgrep`, `poetry`,
   `npm`, etc.). *Recommend:* defer until the detector engine ships
   in v0.5 ŌĆö at that point we have a real reason to require specific
   tools.

7. **Telemetry first-run prompt.** Plan ┬¦0 says "opt-in only".
   v0.3 ships `telemetry.enabled: false` in config but no prompt.
   *Recommend:* keep deferred to v0.8 per the roadmap. *Why:* prompt
   UX needs careful design.
