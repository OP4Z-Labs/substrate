# Substrate v2.0 + Flint v1.0 — Smoke Test Report

> **Date:** 2026-05-15
> **Mode:** Read-only smoke test before publish
> **Verdict:** needs-fixes-before-ship (4 high-severity findings; the §7 headline pipeline itself works end-to-end)

## Executive summary

- **Counts:** 0 critical / 4 high / 4 medium / 3 low
- **Headline result:** the §7 proposal-pipeline scenario works end-to-end. A v2 workflow runs via `substrate run`, writes a session-event-log, drift detection finds the ad-hoc step, the classifier emits a typed proposal, the applicator writes a comment-preserving YAML edit. The walker (`substrate review --proposals`) presents the proposal in dry-run + batch-confirm modes. Comments above, between, and below the steps array all survive the applicator edit.
- **Top 3 blockers (must address before npm publish):**
  1. `op4z-flint-1.0.0.tgz` ships without a `LICENSE` file (package.json declares MIT but no LICENSE file exists in the flint repo). `op4z-substrate-2.0.0.tgz` correctly includes LICENSE.
  2. Multiple v2 CLI commands return exit code 0 even when they print failure messages (`substrate audit` when RULES.yaml is missing; `substrate validate` when manifests fail; `substrate doctor` when summary reports `1 error`; `substrate knowledge show` when KNOWLEDGE.md is missing; `substrate hooks describe <nonexistent>`). This breaks CI gating.
  3. `substrate init` does NOT scaffold the v2 directory layout (`substrate/workflows/`, `substrate/hooks/`, `substrate/doc-checks/`, `substrate/RULES.yaml`). It scaffolds the v1 `auto/` layout only. A consumer who runs `npm i @op4z/substrate && npx substrate init` has no v2 surface to invoke until they manually copy templates from `node_modules/`.
- **Recommended next action:** Fix the four high-severity findings (LICENSE add, exit-code propagation across at least audit/validate/doctor/knowledge-show, init scaffolding gap, and the broken `new-service.yaml` template). The headline pipeline works — the gaps are at the surface, not the substrate.

## What was not tested

- **Real AI session integration:** Cannot drive prompt-type steps in `substrate run` without a Claude Code / Cursor session. Path A (deterministic-only workflow) covered the runtime; Path B (programmatic API) covered the pipeline. The orchestrator's prompt-handling code path was exercised via dry-run only.
- **Multi-day escalation simulation:** `escalate_after` mutates severity based on `firstSeenAt` / `ageDays`, which requires real elapsed time. Not feasible inside a one-shot smoke run.
- **Cross-platform behavior:** Linux only. Windows path-handling and shell-quoting paths not exercised.
- **Real GH Action run:** Action binary parses + invokes correctly, but a full PR-context run would need the action to be reachable on GitHub. Tested only the binary's local execution path.
- **`substrate scheduler --check` triggering invocation:** Scheduler is read-only by design (per the v2 plan §3.8); no `--auto-run` to exercise.
- **`composes_findings_of` freshness warnings:** Requires sub-audit sidecars in `substrate/audits/<wf>-latest.json`. Did not generate sub-audit fixtures with stale timestamps.
- **`substrate review --proposals` interactive prompts:** Only tested `--dry-run` and `--batch-confirm` (non-interactive paths). The accept/reject/edit/defer/skip TUI prompts weren't walked.
- **Stale-proposal aging in doctor:** Requires pending proposals with `generatedAt` older than the staleness window. Not feasible inside the smoke window.
- **Memory-frontmatter doctor check:** Would need a memory fixture with malformed YAML frontmatter. Skipped for time.
- **Plural knowledge-source plugins (kubernetes / env-registry):** Only `docker-compose.yml` + `.env.example` (the v1 path) was exercised. Kubernetes Secret + ConfigMap key-only-no-value pinning was not regression-tested against a Kubernetes manifest fixture.
- **MCP tool calls beyond `initialize`:** Only the handshake was tested; `tools/list`, `tools/call` not exercised.
