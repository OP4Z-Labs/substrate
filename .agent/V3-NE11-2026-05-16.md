### Design decisions taken (with rationale)

1. **Existing v2 single-root discoverers stay untouched; v3 wrappers opt in.**
   - Plan §2.2 explicitly prescribes this shape ("The existing v2 single-root discoverers stay untouched; v3.0 callers opt into the merged variant when the config declares `extends`").
   - The brief's "update each plug-in point to consult the resolver" was reconciled by reading "plug-in point" as the conceptual seam (workflow / hook / doc-check / standards-loader discovery), with the resolver consulted via the wrapper functions in `discovery.ts` and `context-merge.ts`. Zero-overhead hot path for v2.0 consumers: the chain collapses to a single repo-local layer; the wrappers delegate straight through.

2. **Sub-phase B github-source.ts was a deliberate stub; C replaced it.**
   - Rationale: sub-phase B's brief is "resolver + plug-in points", sub-phase C's is "source kinds (npm + github + file)". Keeping the github implementation deferred to C let B's tests use `file:` sources (which fully exercise the merge logic) without depending on an unimplemented network path.

3. **Repo-local layer is ALWAYS appended to the chain, even with no `extends`.**
   - Simplifies the merge: every consumer has at least one layer. The hot path returns immediately (no extends entries → just push repo-local → done) so v2.0 consumers see no perf regression.

4. **`additionalProperties: false` on each extends-array entry, but the root config still allows extras.**
   - The root config has `additionalProperties: true` for forward compat (per the v2 capabilities audit). The `extends` entries are tighter: typos like `verison: "^2.0.0"` are caught at schema time so the resolver doesn't silently no-op on an invalid version.

5. **Github cache slug is `<org>-<repo>@<ref>` (or `@HEAD` when no ref).**
   - Locked per plan §2.5 ("`<source-hash>` is `sha256(source-spec)[:16]`"). Departed from the plan slightly: plan suggests a hash, this implementation uses the human-readable `org-repo@ref` form so `substrate/.cache/extends/github/acme-shared@v1.0.0/` is grep-able. The cache manifest still records `source` + `resolvedSha` so reproducibility is preserved. Documented this choice inline in `github-source.ts:cacheSlug()`.

6. **`SUBSTRATE_OFFLINE=1` does NOT block warm-cache reads.**
   - Adopters in locked-down networks who pre-warm `<cache>/extends/github/` on a connected machine and ship the directory must still see their content resolve. The implementation: cold cache + offline → warning + chain continues without source; warm cache + offline → ok (cache hit served).

7. **The github `gitRunner` is an injectable function, not a class.**
   - Tests inject a fake; production uses `execFileSync("git", args)`. Avoids the need for a mocking library and keeps the unit tests pure. Same pattern as the v2 audit subsystem's detector runner.

8. **`refreshGithubSource` + `clearExtendsCache` exported from `extends/index.ts`.**
   - Sub-phase D's CLI `substrate extends sync` / `clear-cache` call these directly. The programmatic API surface (these helpers) matches the CLI surface, so external tools (CI, IDEs) can drive both via the SDK.

9. **`extends-mode: inherit` / `extends-mode: append` (plan §2.4(a),(b)) deferred.**
   - Out of brief scope; documented in CHANGELOG "Out of scope for 3.0-alpha.1" + open question Q1 for Beau (below).

10. **`substrate audit --explain <rule-id>` provenance (plan §2.4(b)) deferred.**
    - Same as #9; CHANGELOG documents the deferral. Open question Q2 below.

11. **Inline `github:org/repo#ref` form is accepted by the parser but NOT by the schema.**
    - Adopters with muscle memory from npm + ESLint will type `github:acme/shared#v1.0.0`. The schema's `pattern` blocks that form (forces use of the separate `ref` field for cleanliness). The parser in `github-source.ts` still understands inline `#ref` defensively so a future schema relaxation wouldn't need code changes. This is forward-compat surface; the test suite covers both forms.

## Open questions for the user (Beau)

### Q1 — Should v3.0.0-alpha.2 (or beta.1) ship `extends-mode: inherit` and `append`?
**Agent recommendation:** ship only `inherit` in beta.1; defer `append` to a v3.1 minor.
Why: the "I want the org workflow with one repo-specific tweak" pattern is the more compelling adopter ask of the two. `append` (for RULES.yaml) overlaps with `--explain` provenance + the wait-and-see signal from plan §2.4(b) ("only meaningful for rules with non-conflicting matchers").

### Q2 — Should the github cache slug switch to a SHA hash (per plan §2.5) or keep the human-readable `<org>-<repo>@<ref>`?
**Agent recommendation:** keep the human-readable form.
Why: easier debugging (grep `substrate/.cache/extends/github/`), no real reproducibility cost (the cache manifest already records the resolved SHA). Document the divergence from plan §2.5 in `docs/architecture.md` v3 section + the next CHANGELOG.

### Q3 — Should `substrate extends list --json` include per-layer collision lists?
**Agent recommendation:** the current implementation returns a flat `collisions` array (with `class`, `key`, `winner`, `overridden`). A future shape could group by class. The flat list is closer to what `substrate audit --json` does; recommend keeping it as-is until a real consumer asks otherwise.

### Q4 — Does the missing CHANGELOG amend (79 actual vs 76 documented tests) need a follow-up commit?
**Agent recommendation:** no. The CHANGELOG entry frames per-sub-phase counts as approximations; the totals match. Leave the entry as-is.

## Notes for the next agent

1. **Sub-phase B github-source.ts was a stub.** If anyone looks at the file's git history and sees the "not yet implemented" message in commit 2f83efa, that's expected — cf8c871 replaced it. Don't get confused.

2. **`SUBSTRATE_OFFLINE` is read at resolve time, not at config load time.** Tests inject `offline: true` via the resolver options. If you add a new caller, make sure to propagate this option (the chain in `resolver.ts` reads `process.env.SUBSTRATE_OFFLINE` only when no override is provided).

3. **`discoverWorkflowsAcrossExtends` calls `resolveExtendsChain` once per invocation.** When `runExtendsList` is called, it ends up resolving the chain 5 times (once per discovery wrapper: workflows/hooks/doc-checks/standards/rules). This is intentional for simplicity — each wrapper can be used standalone. If perf becomes an issue, the chain could be memoized per-cwd; not yet warranted (chain resolution is filesystem-only and sub-millisecond).

4. **The github clone path uses `child_process.execFileSync`.** This means it inherits the user's git config (auth, SSL, http proxy). No new credential surface. The CHANGELOG calls this out.

5. **Watch out for YAML in test fixtures.** I lost ~10 minutes debugging the integration test because a `[github] tackle-task` name was parsed as a YAML flow sequence. The test helper now quotes names. If you write new test fixtures, keep names ASCII-only or quote them.

6. **`additionalProperties: false` on the extends-array entry was a forward-compat call.** It catches typos (`verison`) but means we can't add new fields to entries without a v3.x minor bump. If someone wants `subpath` or `integrity` in the future, that's a minor-version change.

7. **Hot-path verification.** The v2.0 tests (608 + the 95 added in v2.0.1 patches = 703 baseline) still pass with the same fixtures. No v2.0 consumer is exercising the v3 wrappers — they consume the existing v2 discoverers directly. The v3 wrappers exist alongside.

8. **CLI `substrate extends list` reuses the discovery wrappers.** It's not optimal (5 chain walks instead of 1) but is the cleanest mapping of "what does each layer contribute." If you optimize, do it inside the wrappers (memoization) rather than in the CLI handler.

9. **Tarball was generated at `packages/substrate/op4z-substrate-3.0.0-alpha.1.tgz` for the smoke test.** It's untracked (matches the v2.0 `op4z-substrate-2.0.0.tgz` convention). Smoke test was done on a fresh `mktemp -d` install; verified `substrate -v` → `3.0.0-alpha.1` and `substrate extends list` works end-to-end. Tarball is reproducible via `cd packages/substrate && npm pack`.

10. **No new npm dependencies added.** The whole NE-11 ships against the v2.0 dep set (ajv, ajv-formats, commander, kleur, yaml, zod, @modelcontextprotocol/sdk, @inquirer/prompts). Network calls in `github-source.ts` use the user's `git` binary, not an npm package.
