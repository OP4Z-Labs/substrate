### Design decisions taken

1. **Templates-history shipping format.** `templates-history/<version>/`
   subdirs mirror the current `templates/` layout. Looked up by the
   manifest entry's `templateVersion`. Missing snapshots return null
   from `getHistoricalTemplate()`; upgrade gracefully falls back to
   v0.5 two-way. **Locked in `packages/cadence/src/util/paths.ts`
   `getHistoricalTemplate()` head comment.**

2. **Monorepo: npm workspaces alone (no Turborepo).** Rejected Turborepo
   as added tooling overhead for v0.8 — task pipelining isn't yet a
   pain point, and `--workspaces --if-present` covers the build/test/
   lint fan-out. **Recorded in the monorepo commit message.**

3. **MCP tool surface: conservative read-only.** Seven tools exposed;
   `init` / `add` / `apply` / `task create|update|complete` /
   `workflow start` deliberately NOT exposed. Rationale: writes mutate
   the user's repo and should stay CLI-driven until MCP grows a
   confirmation convention. **Locked in head comment of
   `packages/cadence/src/commands/mcp.ts`.**

4. **MCP bridge filename: `cadence-server.json` (not `cadence.md`).**
   The other bridges scaffold markdown slash-command files; MCP needs
   a JSON snippet the user copies into their host config. Introduced
   `bridgeFiles(bridge)` helper to return per-bridge file lists.

5. **MCP bridge directory: `.cadence/mcp/` (not `.claude/...`).** MCP
   isn't Claude-specific. Hosts (Claude Desktop, Continue, Cline) all
   read the same registration shape; the bridge file lives in a
   neutral location.

6. **Action JS: zero deps + ESM.** Use the file-based `$GITHUB_OUTPUT`
   API + INPUT_* env var convention instead of `@actions/core`. ESM
   because root package.json is `type: "module"`. Pure node stdlib
   keeps the bundle simple — no webpack/ncc step required.

7. **`fail-on` enforcement at action layer.** Cadence CLI exits 0 for
   "audit ran"; the action interprets findings and decides pass/fail.
   `fail-on: none` makes the action report-only; `error` / `warning`
   fail on non-zero exit. Forward-compatible naming for v1.0 severity
   axis.

8. **Adapter contract duplicated inline (3 reference adapters).**
   Continuing the v0.5 stub-adapter pattern: peer packages don't
   import from cadence at build time. Structural typing at runtime
   via `isTaskAdapter()`. Tests inject narrow client-like fakes
   (LinearClient, JiraClientLike, OctokitLike) — no real HTTP, no
   credentials needed.

9. **Linear adapter: lazy default-export instantiation.** Module-level
   `_instance` only constructs `LinearClient` on first method call,
   so import-time loads can probe the package without credentials.
   Same pattern for Jira + GitHub adapters.

10. **Linear `completeTask`: first workflow state with type=completed.**
    If a team has multiple completed states (Done + Cancelled both
    type=completed) the adapter picks whichever Linear returns first.
    Callers needing precision use `updateTask --status "<NAME>"`.

11. **Jira adapter: ambient `jira-client.d.ts` declaration.**
    jira-client doesn't ship types; `@types/jira-client` isn't
    maintained. Ships a narrow ambient module declaration covering
    only the constructor + 6 methods we call.

12. **Jira `updateTask`: status updates are workflow transitions.**
    The adapter looks up the issue's available transitions and
    matches by name (case-insensitive). `completeTask` matches
    `Done` / `Closed` / `Resolved` / `Complete` via regex.

13. **Jira assignee writes: Server semantics only at v0.8.** Cloud
    needs `accountId`, Server uses `name`. v0.8 always sends `{ name }`.
    Cloud users may need to wrap until v1.0 adds host-kind detection.

14. **GitHub adapter task ID: `owner/repo#number`.** Cadence task IDs
    must be globally unique. GitHub issues are repo-scoped, so the
    ID embeds the owner/repo. Numeric IDs work when GITHUB_OWNER +
    GITHUB_REPO env vars are set.

15. **GitHub `status` round-trip.** GitHub state is binary (open/closed);
    `state_reason` (completed / not_planned / reopened) is promoted
    into the cadence `status` string on closed issues.

16. **GitHub `updateTask` labels: full replacement.** Matches GitHub's
    PATCH /issues/{n} semantics. Priority label updates strip-then-
    append to avoid duplicate `priority:*` labels.

17. **Astro 5.x (not 6.x).** Astro 6 requires Node 22+; cadence's
    engines are Node 20+. Astro 5.18.1 is latest in the 5.x line and
    keeps the engines aligned.

18. **Docs site: dogfood-scaffolded with cadence.** `cadence init`
    run inside `docs-site/` so the auxiliary site is itself a
    cadence-managed project. Means any audit/standards
    improvements made to cadence flow through the docs site too.

19. **Telemetry: synchronous init + after-run emission.** init stays
    sync; the opt-in is a one-time NOTICE flipping `prompted=true`,
    not an interactive prompt. Interactive prompt deferred to v1.0
    when `runInit` can become async. Events emit AFTER `parseAsync`
    completes — so the first `cadence config --telemetry on`
    successfully records its own event.

20. **Telemetry event shape (locked v0.8 public API):**
    `{ v: 1, ts, cadenceVersion, osFamily, command, audit?, errorType? }`.
    **No project paths, no user identifiers, no rule body content,
    no audit findings, no message bodies.** errorType is the class
    name only. Recorded in head comment of `util/telemetry.ts` and
    pinned by the "forbidden fields" unit test.

21. **`dist/action/index.js` checked into git.** `.gitignore` rewritten
    with `**/dist/` blanket ignore + `!/dist/action/` un-ignore so
    the Action entrypoint commits while per-package build outputs
    stay out of the tree.

22. **`config` is no longer deferred.** v0.8 ships `cadence config
    --telemetry on|off` + a no-flag print. `--enable / --disable /
    --eject` remain deferred to v1.0; `review` and `standards`
    remain deferred (still exit 2 with hint).

## Open questions for the user

1. **Should `cadence init` become async for v1.0 to support a real
   interactive opt-in prompt?** v0.8 ships a one-time NOTICE banner
   because making init async would break existing sync callers (tests,
   programmatic users). v1.0 could split into `runInit` (sync, used by
   tests/programmatic callers) and `runInitInteractive` (async, used by
   the CLI when stdin.isTTY).
   *Recommend:* defer. v0.8's banner is honest about telemetry being
   off-by-default; an interactive yes/no on top adds little signal.
   *Why:* the surface that benefits (CI / scripted users) already
   knows what they want; interactive humans are well-served by reading
   the banner + running `cadence config --telemetry on` if interested.

2. **MCP host registration format: should cadence emit per-host
   variations?** v0.8 ships the Claude-Desktop-shaped JSON. Continue
   and Cline use slightly different config shapes (Continue's lives
   in `.continue/config.json`, Cline reads from VSCode settings).
   *Recommend:* hold at Claude-Desktop format for v0.8; document the
   Continue / Cline variations in the README v1.0 work.
   *Why:* Claude Desktop is the most common MCP host today; the
   registration shape happens to be the simplest. Users on other
   hosts copy the `command` + `args` + `env` into their host's
   format with one substitution.

3. **GitHub Action versioning vs cadence versioning.** v0.8 hard-codes
   `cadence@<version>` from the user's input (default "latest"). When
   cadence ships v0.8.1 with a CLI bug fix, action consumers pinning
   `BeauGoldberg/cadence@v0.8.0` still get the new CLI via
   `cadence@latest`. Is this the right precedence?
   *Recommend:* keep as-is for v0.8. Document that the action's
   `version` input is the source of truth for the CLI version; the
   action tag is the action-wrapper version.
   *Why:* action wrappers rarely change; CLI patches benefit from
   auto-update.

4. **Adapter publication strategy at v1.0.** All three reference
   adapters declare `"private": true` at v0.8. At v1.0 publication
   time, version-coupling questions surface (does
   `@cadence/adapter-linear@1.0.0` support `cadence@1.0.0` only, or
   the latest cadence at install time?).
   *Recommend:* publish adapters with `peerDependencies:
   "cadence": "*"` and structural typing as the runtime contract.
   The TaskAdapter contract is locked in cadence's
   `src/extensions/task-adapter.ts`; adapter packages don't import
   from cadence so peer-dep range is widely tolerated.
   *Why:* this matches the pattern most plugin ecosystems use; the
   structural-typing escape hatch means version-skew is rarely a
   real concern in practice.

5. **Astro 6 migration when Node 22 becomes the engines floor.**
   Pinned to Astro 5.x for v0.8 because Astro 6 requires Node 22.
   *Recommend:* track. When the OP4Z auxiliary-site family migrates
   to Node 22, bump cadence docs-site to Astro 6 in lockstep.
   *Why:* docs sites have no first-party reason to push the node
   floor; let the broader ecosystem drive.

6. **Telemetry endpoint design for v1.0.** Options: bring-your-own
   collector (env var pointing at a URL), shipped collector (free
   for OSS use, optional self-host), or anonymized GitHub commit
   from cadence-org repo.
   *Recommend:* bring-your-own. Cadence ships the event shape +
   the endpoint POST helper; users pick the collector. Avoids
   ownership / privacy questions about a centralized
   cadence-collector.
   *Why:* mirrors the plugin contract pattern. The framework
   provides the surface; the user chooses the implementation.

7. **Three-way merge UX for terminal width.** v0.8's three-pane diff
   ("your edits" + "cadence changes" + "raw drift") emits all three
   sequentially. On narrow terminals this is dense. Should v1.0
   add a side-by-side or interactive selector?
   *Recommend:* not yet. The diff lines are already standard
   unified-diff shape; piping through `bat` or a paged
   viewer handles wide terminals fine. Interactive selectors are a
   classification problem (which hunks go where) and add tooling
   surface.

### Monorepo layout details v1.0 publish work will depend on
- `packages/cadence/` is the publishable CLI (`cadence` on npm).

  flip to `false` at v1.0 publish time.

- Root `package.json` is `"private": true` (the workspace manager
  isn't a publishable artifact).
- `docs-site/` is a workspace with its own `package.json`. Astro is
  the only runtime dep. Treat it as build-only — never publish.

### Adapter API conventions established
- **Structural typing inline.** Each adapter package duplicates the
  TaskAdapter interface in its own source file. NO build-time
  import from cadence.
- **Narrow client-like interface for tests.** The pattern is: declare
  a `<Vendor>ClientLike` interface listing only the SDK methods the
  adapter actually calls. Tests inject a fake implementing just those.
- **`createXxxAdapter(options)` factory + lazy default export.**
  Module-level `_instance` only constructs the SDK client on first
  method call.
- **`completeTask`** is whatever-makes-sense-for-the-tracker. Linear:
  first state with type=completed. Jira: first transition matching
  Done/Closed/Resolved/Complete. GitHub: close with
  state_reason=completed.
- **Mapping discipline.** Each adapter's README has a `Mapping` table
  documenting how cadence fields land in the tracker's data model.
  Maintain this table; it's the user-facing contract.

### Three-way merge corner cases discovered
- **`templates-history/<version>/` lookup is best-effort.** When a
  user scaffolded against a cadence version that this build doesn't
  ship history for (e.g. v0.6 — never released, never snapshotted),
  the upgrade falls back to two-way. The classifier's
  `originalTemplatePath` is `null` in that case and
  `threeWay: false`. Renderer signals the fallback in the header
  banner ("merge mode: two-way") and on each modified-entry diff.
- **Going forward, every shipped minor version must snapshot
  `templates/` into `templates-history/<NEW_VERSION>/` BEFORE the
  version bump.** v0.5.0 was retroactively snapshotted (from git
  HEAD at v0.5 time, identical to working tree). v0.8.0 going
  forward will need a `templates-history/0.8.0/` snapshot for any
  v1.0+ build that wants real three-way merge on 0.8-scaffolded
  manifests.
- **Conflict-marker block style.** `.cadence-merge` file uses
  git-style markers when three-way is available. Some editors
  (VSCode, Helix) recognize these for visual merge; others (Vim)
  don't. v0.8 punts on emitting a non-marker variant.

### Gotchas / conventions established
- **Don't use `require()` in test files.** ESLint rejects; use
  top-of-file `import` statements. Pattern was already locked in
  v0.5; the telemetry tests caught one slip.
- **Telemetry must NEVER throw.** Errors in emission are swallowed.
  See `emitTelemetryEvent` in `util/telemetry.ts`. Telemetry is
  optional; it must not gate the user's command.
- **MCP captured stdout via `process.stdout.write` swap.** Tactical
  not structural — see head comment of `commands/mcp.ts`. Don't
  refactor every command to return strings; that's a much larger
  change for marginal gain in v1.0.
- **Adapter package tests aren't wired into `cadence-monorepo`'s
  lint or typecheck scripts.** Only the cadence workspace has
  those (the adapters' own package.json doesn't declare them).
  This is deliberate at v0.8 — adapters are private; v1.0 should
  add lint/typecheck to each adapter's scripts.
- **GitHub Actions security note** — `dist/action/index.js` is what
  consumers fetch. Never accept untrusted input from the
  triggering event (issue.title, etc.) into shell commands. The
  action uses `spawnSync` with `args` array (not `shell: true`),
  which prevents shell injection from the user-supplied `command`
  input.

### Patterns the next agent should follow
- **One commit per logical batch.** Suggested boundaries are
  listed in the brief — the 10 commits here match that pattern.
- **Gates after each commit.** `npm run build`, `npm run lint`,
  `npm run typecheck`, `npm test`. v0.8 stayed gate-green
  throughout.
- **Verify dep versions via `npm view <pkg> version` BEFORE
  installing.** v0.8 added 6 new deps (MCP SDK, zod, Astro,
  Linear SDK, jira-client, Octokit). Each was version-checked
  first.
