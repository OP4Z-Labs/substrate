## What v2.0 is

v2.0 layers an **AI-orchestrated workflow runtime** + a
**self-reinforcement loop** on top of v1.0's audit detector + standards
foundation. Eleven primitives total:

| #   | Primitive                          | Phase | Headline value                                                                                                 |
| --- | ---------------------------------- | ----- | -------------------------------------------------------------------------------------------------------------- |
| P1  | Workflow manifest                  | B1    | Declarative YAML + body for every recurring workflow                                                           |
| P2  | Two-layer architecture             | B1    | Deterministic primitives below, AI orchestration above; no cross-layer reimplementation                        |
| P3  | Cross-cutting hooks                | B2    | "Every workflow does X" without per-workflow copy-paste                                                        |
| P4  | Conditional doc-check registry     | B2    | "If commit msg matches, require changelog" as a deterministic gate                                             |
| P5  | First-class memory                 | B2    | Read Claude Code's memory by default; filter by type/scope/tags/file-overlap; inject into workflow context    |
| P6  | composes_findings_of               | B2    | Cross-workflow dependency; pull findings from upstream workflows into context                                  |
| P7  | escalate_after                     | B2    | Age-based severity escalation on RULES.yaml entries                                                            |
| P8  | trigger: schedule                  | B3    | cron / interval / every-N-commits triggers; non-invasive scheduler primitives                                  |
| **P9** | **Proposal pipeline (the headline)** | **B3**  | **Drift detectors → typed proposals → applicators → review walker. Substrate watches its own runs.**         |
| P10 | substrate doctor enhancements      | B3+B4 | Five v2 checks (memory-frontmatter, rules-doc-coverage, workflow-coverage, stale-proposals, escalation-debt)   |
| P11 | Plural knowledge sources           | B4    | Typed plugin contract; docker-compose + kubernetes + env-registry built-in; custom plugin support              |

The headline is **P9 — the proposal pipeline.** Every workflow run
emits a session-event-log. Drift detectors observe what the workflow
declared vs what actually happened. Findings classify into eight typed
proposals (`add-to-workflow-step`, `add-to-memory`,
`strengthen-context-load`, `add-to-rule`,
`add-to-doc-check-registry`, `add-to-standards-doc`,
`cross-link-existing`, `add-to-adr`). A walker (`substrate review
--proposals`) presents them for accept / reject / edit / defer / skip.
Accepted proposals invoke deterministic applicators that write the
change back to disk (workflow YAML edits use comment-preserving
helpers; RULES.yaml gets appended; memory writes use B2's storage
discovery; ADR drafts auto-increment DEC-XXX).

The framework that ships your conventions **and** watches itself drift
away from them. That's the v2.0 thesis.

## How v2.0 was built (phase arc)

### Phase B1 — Foundation (~2 weeks of agent work)

Set up the workflow manifest schema, the two-layer architecture, the
discoverer + context loader (pure deterministic modules), and three
reference workflow templates (`tackle-task`, `audit-service`,
`audit-package`). Exit signal: `substrate run <workflow>` works end-to-
end against the reference templates; `substrate validate` works
against arbitrary manifests; deterministic primitives layer is
invokable from CI / scripts.

5 commits. ~88 new tests.

Key design decisions documented in `HANDOFF-2026-05-15-v2-phaseB1.md`:

- Manifest schema is opinionated but extensible — every field
  documented in `schemas/workflow.schema.json` + the prose plan §4.
- Two-layer split is the load-bearing architecture decision: every
  primitive that ships ever after either lives in the deterministic
  layer (validators, query commands, applicators) or the
  orchestration layer (run-command, hook dispatch, session-event
  emission).
- Reference workflow templates are working starting points, not
  examples. They demo all the manifest fields that matter.

### Phase B2 — Reinforcement (~1.5 weeks)

Added the cross-cutting hook system, conditional doc-check registry,
first-class memory integration, composition, and escalation. Exit
signal: hooks fire correctly; memory query works against Claude
Code's directory; composition + escalation work in reference
workflows.

2 commits. +141 new tests.

Key design decisions documented in `HANDOFF-2026-05-15-v2-phaseB2.md`:

- Memory backend discovery follows a fixed precedence: flag → env →
  config → Claude Code default → none. Substrate ships zero memory
  backend of its own; it reads from whatever's there. This is the
  "Claude-Code-path-bridge" approach from plan §6.
- The hook dispatch model is fail-loud-fail-fast. A hook that can't
  fire (missing handler) returns `status=skipped` with a clear
  message; it never silently succeeds.
- `composes_findings_of` requires a recency window (`require-fresh-
  within: 24h`) to prevent stale findings polluting downstream
  context.
- `escalate_after` mutates `severity` in-place and stamps
  `originalSeverity` + `firstSeenAt` + `ageDays` so reviewers see
  the full history.

### Phase B3 — Loop closure (~2 weeks)

The biggest phase. Shipped the headline primitive — the proposal
pipeline — plus the schedule trigger and the `weekly-proposal-walk`
reference workflow. Exit signal: drift detection works on the plan
§7 worked example; `substrate review --proposals` walks; applicators
write correct files.

2 commits. +127 new tests.

Key design decisions documented in `HANDOFF-2026-05-15-v2-phaseB3.md`:

- Session-event-log is a **separate channel** from
  `~/.config/substrate/telemetry.log`. Telemetry stays at `v: 2`;
  session-event logs are self-describing via the event discriminant
  (no version field).
- Sanitiser blocklist (no `/home/`, no `Bearer`, no `ghp_`, no emails)
  applies to every string field. Truncation at 120 chars on
  `description` / `prompt` / `output`.
- Six built-in drift detectors:
  `adhoc-step` / `skipped-step` / `out-of-order` / `context-gap` /
  `repeated-prompt` / `rule-violation-recurrence`. Each emits
  findings with confidence tiers (low / medium / high). Recurrence
  escalates confidence.
- Proposal ID is deterministic — `sha256(kind | workflowId |
  driftSignature)`, truncated to 12 hex. Same drift → same ID →
  queue de-duplicates.
- In-house cron parser (5-field with `*`, ranges, comma-lists, step,
  named days/months). ~50 LOC. No new dependency. Unsupported cron
  forms fall through to "always due" with a warning (loud failure
  mode).
- Applicators are deterministic; AI polish (drafting standards-doc
  prose, refining ADR templates) is a future B5+ enhancement. The
  applicators write usable drafts with embedded
  `<!-- substrate-proposal: <id> -->` anchors.
- Comment-preserving YAML edits cap at depth 2. Deeper edits would
  require a parser-aware model — added when needed.
- `substrate scheduler --check` lists, never runs. Invocation is the
  consumer's job (CI / cron / AI session — three patterns
  documented in `docs/scheduling.md`).

### Phase B4 — Polish + release (~1 week)

Shipped the doctor v2 checks, plural knowledge sources, the
`substrate query sessions` CLI wrapper deferred from B3, the docs
site v2.0 update, and the release prep (CHANGELOG + migration
guide + release checklist + version bump + npm pack verification).

5 commits. +35 new tests.

Key design decisions documented in `HANDOFF-2026-05-15-v2-phaseB4.md`:

- Doctor v2 checks each addressable individually via `--check
  <name>`; scoped mode suppresses the v1 baseline so output matches
  the requested slice.
- Plural knowledge sources via `substrate/knowledge-sources.yaml`
  manifest. Built-in plugins (docker-compose / kubernetes /
  env-registry); custom plugins via `registerKnowledgePlugin()`.
  Glob expander is minimal (just `**` + `*` + literal). v1 fallback
  preserved.
- Kubernetes Secret + ConfigMap surface KEYS ONLY. Values never leave
  the filesystem. Test pin.
- README.md + LICENSE copied into `packages/substrate/` so the npm
  tarball picks them up.
- Templates-history/2.0.0 snapshot committed for future three-way
  merge support.

## v2.0 design principles (locked)

These are the design decisions every primitive obeys. Future v2.x /
v3.x agents should not re-litigate them without writing an ADR.

1. **AI-as-orchestration-runtime; deterministic primitives below.**
   Plan principle 2.1. Substrate ships zero AI of its own. The
   orchestration layer drives whatever AI session the consumer has
   (Claude Code by default, Cursor / MCP optionally). Deterministic
   primitives never touch AI.

2. **Cross-cutting hooks AND per-workflow prose followups.** Plan
   principle 2.2. Hooks are the "every workflow does X" surface;
   followups are the "this specific workflow says X" surface. Both
   compose.

3. **Memory is a first-class context source.** Plan principle 2.3.
   Workflows declare what memory to load; the orchestrator injects
   it into the AI prompt. Memory backend is the user's (Claude Code
   by default).

4. **Conditional doc-check registry ships in v2.0.** Plan principle
   2.4. The registry is the "if X, require Y" surface — used as a
   deterministic gate.

5. **Engine ships the contract; users bring the content.** Plan
   principle 2.5. Substrate's `init` scaffolds the layout; the user
   fills in the prose. v2.0 ships reference templates as worked
   examples, not authoritative content.

## v2.0 → v2.x roadmap (deferred but not forgotten)

These items were called out as deferrable in plan §13 (out of scope)
or as B4+ polish in the phase HANDOFFs:

- **AI-drafted applicators.** B3's standards-doc + ADR applicators
  write deterministic drafts with embedded anchors. A future minor
  could wrap them in an orchestrator-layer AI polish step.
- **`substrate scheduler --auto-run`.** Today's scheduler is
  non-invasive. A future minor could let cron-driven invocations
  call `substrate run` directly.
- **Public extension point for the noop-handler registry.** B2's
  open question; B3/B4 stayed internal. Recommend leaving until a
  real plugin contract emerges from consumer demand.
- **Visual / web UI for proposal review.** Plan §13 scopes this
  out of v2.0. CLI walk is the contract.
- **Substrate-native memory backend.** Plan §13 scopes this out.
  v2.0 reads Claude Code's existing format. A future major
  (v3.0) might add one.
- **Custom proposal types beyond the 8 listed.** Plan §13 scopes
  this out. Future-extensible via plugin contract, but v2.0 ships
  the eight.
- **Beau-hosted services / SaaS.** Plan §13 scopes this out
  permanently for v2.x. v2.0 stays local-log-only.

## Why v2.0 matters

v1.0 made substrate the framework you scaffold once and edit forever.
v2.0 makes substrate the framework that learns from its own runs.

Every other automation tool defines its conventions and never looks
back. Substrate's proposal pipeline reads the session-event-log of
every workflow run, detects the deltas between declared and actual
behaviour, and proposes typed changes back to the conventions
themselves. The framework that watches itself drift.

That's the moat. The substrate "engine ships the contract; users
bring the content" principle (plan 2.5) means every consumer accumulates
their own opinionated content over time. v2.0's proposal pipeline
means that accumulation is observable, reviewable, and continuous.

Ship it.
