## Summary

Folded the six targeted improvements from the v3 exploration report
into v2.0 itself. v2.0 hadn't published, so closing these gaps before
publish (rather than queueing for a v2.1 patch) keeps consumers on a
single coherent surface. The TI changes are fully additive — v1.0 and
B1/B2/B3/B4 surfaces remain operational.

The single biggest piece is **TI-1 — the AI step engine**. The
orchestrator's `runStep` had returned `status: "deferred"` for six
step types since B1; B2 was supposed to ship the handlers and only
shipped the dispatch skeleton. TI-1 closes that gap. The proposal
pipeline (B3's headline) now sees real `prompt-issued` / `step-confirm`
/ `adhoc-step` events from sessions where they occur — not just from
hand-crafted test fixtures.

## Per-item status

### TI-1 — Finish the AI-step engine

**Commit:** `43a3903 feat(v2): step engine — runs all six AI-step types end-to-end [TI-1]`

**Files:**
- `packages/substrate/src/v2/orchestrator/step-handlers.ts` (new, 487 lines) — handler module
- `packages/substrate/src/v2/orchestrator/transport.ts` (new, ~80 lines) — `OrchestrationTransport` interface + `NO_OP_TRANSPORT`
- `packages/substrate/src/v2/orchestrator/run-command-types.ts` (new, ~20 lines) — shared `RunStepResult` type (breaks circular dep)
- `packages/substrate/src/v2/orchestrator/run-command.ts` — dispatches to step handlers; `runStep` no longer returns `deferred` for any AI step
- `packages/substrate/tests/v2-step-handlers.test.ts` (new, 22 tests)
- `packages/substrate/tests/v2-run.test.ts` — flipped the `deferred` assertion to assert the new ok-in-no-transport flow
- `packages/substrate/tests/v2-reference-templates.test.ts` — `tackle-task` now runs prompts end-to-end; halts only at missing sub-workflows

**Design decisions:**
1. **No-transport mode is the default.** Without an attached
   `OrchestrationTransport`, the engine emits prompts as session events,
   defaults responses to `null`, auto-approves gates. This keeps the
   engine deterministic for tests and CI. Transports are the seam for
   AI surfaces (Claude Code, Cursor, MCP).
2. **Sub-workflow depth cap is 5.** `MAX_SUB_WORKFLOW_DEPTH=5` prevents
   infinite recursion. Sub-workflows get their OWN session log; the
   parent records the child's session-log path on the step output.
3. **`prompt-and-action` always emits step-confirm.** Tree-mutating
   actions deserve an audit-trail confirmation regardless of
   `must-confirm` being explicitly set.
4. **`gate` has two evaluation modes.** `must-confirm: true` goes
   through the transport (auto-approves without). Otherwise: walks
   the manifest's `acceptance.required-steps` and checks each prior
   step result.
5. **`propose-doc-change` routes through the proposal queue.** Writes
   to `substrate/proposals/pending/` rather than mutating the doc
   directly. Same walker (`substrate review --proposals`) applies it.
6. **Defensive validation belt + braces.** The schema requires
   `prompt` on prompt-type steps and `workflow` on invoke-sub-workflow;
   the handler's defensive checks are the second line of defense for
   programmatic callers that bypass schema validation.

**Smoke:**
- `substrate run tackle-task --dry-run` walks all 8 steps as skipped.
- `substrate run tackle-task` (no transport) runs steps 1-4 to `ok`,
  fails cleanly at step 5 (`tests` → missing sub-workflow
  `run-tests-scoped`).
- Session log contains `prompt-issued` + `step-confirm` events for the
  prompt + must-confirm steps; proposal pipeline picks them up
  (`auto-drift-detect: 3 findings`).

**Acceptance:** all six step types execute, deferred assertions
flipped, `tackle-task` runs end-to-end, "requires B2 step engine"
message removed everywhere, new handlers tested.

### TI-2 — Wire `session-start`, `session-end`, `file-change` hook triggers

**Commit:** `2257014 feat(v2): wire session-start, session-end, file-change hook triggers [TI-2]`

**Files:**
- `packages/substrate/src/v2/deterministic/watch-command.ts` (new, ~170 lines) — long-running watcher
- `packages/substrate/src/v2/orchestrator/run-command.ts` — dispatch session-start before workflow-start; dispatch session-end after workflow-completion
- `packages/substrate/src/cli.ts` — new `substrate watch [path]` command
- `docs/architecture.md` — lifecycle hooks table
- `packages/substrate/tests/v2-lifecycle-hooks.test.ts` (new, 9 tests)

**Design decisions:**
1. **No new runtime dep.** Used `node:fs.watch` with `recursive: true`
   (Node 20+; substrate's `engines.node` is already `>=20.0.0`).
   `chokidar` would have been more portable but adds dependency
   weight; substrate's no-runtime-deps story matters more here.
2. **100ms debounce per path.** Editors emit double events on save
   (one for the temp file, one for the rename); dropping back-to-back
   events for the same path keeps hook dispatch sane.
3. **Implicit exclude list.** `substrate/sessions/`,
   `substrate/proposals/`, `.git/`, `node_modules/`, `dist/` are
   excluded. Watching the sessions dir while a workflow runs would
   create a feedback loop.
4. **Foreground watcher, no daemon.** `substrate watch` blocks until
   SIGINT / SIGTERM. `--max-events <n>` provides a graceful exit for
   CI / smoke. Daemonisation is the consumer's choice (systemd,
   tmux, etc.).
5. **session-start fires only on real runs.** `--dry-run` skips
   session-start / session-end / workflow-* hooks. The session log
   itself is still written so dry-run remains observable.

**Smoke:** Live watcher fires on a file save inside the watched tree;
session-start + session-end hook records appear in `hookRuns`.

**Acceptance:** four lifecycle triggers documented, watcher works,
9 tests cover trigger order + dry-run skip + path exclusions.

### TI-3 — `substrate explain <workflow>`

**Commits:** `cca7ac5 feat(v2): substrate explain + scheduler --auto-run [TI-3 TI-5]` (combined with TI-5)

**Files:**
- `packages/substrate/src/v2/deterministic/explain-command.ts` (new, ~270 lines)
- `packages/substrate/src/cli.ts` — `substrate explain <id>` command
- `docs/architecture.md` — explain in commands table
- `packages/substrate/tests/v2-explain.test.ts` (new, 6 tests)

**Design decisions:**
1. **Uses `loadContext` directly.** Same path the orchestrator's
   `runV2Workflow` takes for context resolution. Guarantees parity
   between "what explain shows" and "what run loads".
2. **No template rendering.** Step prompts ship as-is; explain does
   not substitute `${id.output}` placeholders because there are no
   prior outputs at explain time.
3. **`--for-files` is a passthrough.** Forwards to
   `loadContext({ tree.changedFiles })` for the `intersect-with-changed-files`
   memory filter.
4. **Both human and JSON output.** Human output trimmed to 4-line
   step prompt previews + 500-char memory injection preview; `--json`
   emits the full structured envelope.

**Smoke:** `substrate explain tackle-task --json` emits the full
context + 8-step structure.

**Acceptance:** all 3 primary reference workflows + 5 new ones
explain correctly, `--json` works, missing workflow returns exit 2.

### TI-4 — Replace ripgrep look-around regexes in shipped RULES

**Commit:** `5b0c8c8 fix(v2): replace ripgrep look-around regexes in shipped RULES [TI-4]`

**Files:**
- `packages/substrate/templates/standards/cross-cutting/detectors/be-apiv-001.mjs` (new)
- `packages/substrate/templates/standards/cross-cutting/detectors/fe-ts-001.mjs` (new)
- `packages/substrate/templates/standards/cross-cutting/detectors/xcut-md-001.mjs` (new)
- `packages/substrate/templates/standards/cross-cutting/RULES.yaml` — three rules switched to `type: script`
- `packages/substrate/src/commands/doctor.ts` — new `ripgrep-lookaround` check
- `packages/substrate/src/cli.ts` — updated `--check` help text
- `packages/substrate/tests/v2-ripgrep-lookaround.test.ts` (new, 10 tests)

**Design decisions:**
1. **Scripts over `--pcre2`.** v3 exploration suggested auto-detecting
   `--pcre2` availability and passing it when needed. We went the
   other way: rewrite the offending rules as script detectors so the
   shipped RULES works on every ripgrep install (incl. Alpine /
   minimal CI images where ripgrep is built without PCRE2 support).
2. **Doctor warns on consumer look-arounds.** The new
   `ripgrep-lookaround` check scans the consumer's RULES.yaml for
   `(?=`, `(?!`, `(?<=`, `(?<!` in ripgrep detectors and surfaces a
   warn-severity entry. Catches the gotcha at audit time rather than
   at first audit run.
3. **`substrate init` scaffolds the detectors.** Without copying
   `substrate/standards/cross-cutting/detectors/*.mjs` alongside
   `RULES.yaml`, a fresh init would have three broken rules. Init
   now copies the detectors dir explicitly.

**Affected rules:**
- `BE-APIV-001` — was `(?!api/v)`; now a script that walks `.py`
  files + flags FastAPI/Flask routes not under `/api/v<N>`
- `FE-TS-001` — was `(?!\s*//\s*(eslint|TODO|HACK))`; now a script
  that flags `: any` without an eslint/TODO/HACK/FIXME/XXX trailing
  comment
- `XCUT-MD-001` — was `^---\n(scope|area):` (multiline without
  `multiline: true`); now a script that parses frontmatter properly

**Smoke:** Each script detector tested against a fixture with both a
matching and non-matching file; verified the script catches what its
rule claims to detect.

**Acceptance:** zero look-around patterns in shipped RULES; doctor
check warns on consumer look-arounds; init scaffolds the detectors.

### TI-5 — `substrate scheduler --auto-run`

**Commit:** `cca7ac5 feat(v2): substrate explain + scheduler --auto-run [TI-3 TI-5]` (combined with TI-3)

**Files:**
- `packages/substrate/src/v2/deterministic/scheduler-command.ts` — new `runSchedulerAutoRun` export
- `packages/substrate/src/cli.ts` — `--auto-run`, `--workflow <id>`, `--batch` flags on the existing `scheduler` command
- `docs/architecture.md` + `docs/migration-from-1.x.md`
- `packages/substrate/tests/v2-scheduler-auto-run.test.ts` (new, 8 tests)

**Design decisions:**
1. **Lazy import of `runV2Workflow`.** Keeps the deterministic
   `--check` path dep-light. The orchestrator only loads when
   `--auto-run` is actually invoked.
2. **No state-tracking on top of `recordWorkflowRun`.** The
   orchestrator's existing call into `recordWorkflowRun` (which runs
   on every workflow with `trigger: schedule`) is what updates
   `scheduler/state.json`. `--auto-run` is just a loop over `due`
   that calls the orchestrator; no parallel bookkeeping.
3. **Failures don't crash the loop.** A failed workflow in the auto-
   run set records `ok: false` and the loop continues. The final
   exit code reflects whether ANY workflow failed.
4. **`--batch` is documentation, not behavior.** `auto-run` always
   exits after firing all overdue; the flag is for callers who might
   otherwise expect a daemon-like mode.

**Smoke:** `substrate scheduler --auto-run --workflow audit-security`
on a fresh fixture fires the workflow + updates scheduler state.

**Acceptance:** auto-run fires due workflows; `--workflow <id>`
restricts to one; state updates correctly; failed workflows surface
as `ok: false`; `--json` envelope.
